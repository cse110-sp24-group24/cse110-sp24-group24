## Discussion Points & Notes

**1. README & Project Structure Walkthrough**

- Sam shared his screen and walked the group through the README in the docs directory to get everyone on the same page.
- A placeholder project name generated by AILA during an earlier prototype session was noted and removed from the repo during the meeting.

**2. Tech Stack Decision (ADR Document 1)**

- Sam presented the three options outlined in ADR Document 1.
- Option 1: plain HTML, CSS, and JS files using ES modules. Option 2 and Option 3 both involve more complex backends and non-trivial hosting setups that the team has less experience with.
- Sam recommended ES modules as the clearest path forward given the 3-week timeline and the team's familiarity with the approach.
- Nishant agreed, favoring simplicity. No objections from the group.
- Key advantage called out: ES modules are modular and interchangeable, making the codebase easy to organize and edit incrementally, which is especially important when vibe coding because the professor and TAs will be reading through it.
- **Decision:** ES modules confirmed. ADR Document 1 to be finalized.

**3. Sprint Overview & GitHub Issues**

- Sam shared a high-level sprint overview document he created in the 30 minutes before the meeting, derived from the team's user stories.
- 23 total tasks split across 3 sprints. Sprint 1 has 8 issues, all of which are already created in the GitHub Issues tab.
- Issues are numbered and should be read bottom to top in the GitHub UI (filter resets on each page open).
- Each issue is clearly documented, including what it creates, which user stories it draws from, and what it depends on. The "depends on" column is important: some issues cannot be completed until a prerequisite issue is done.
- Sam asked the group to read through all the Sprint 1 issues on their own time before assignments are made, and to note any preferences for what they want to work on.
- Itai suggested all 7 members look through the issues and flag preferences, which will be factored into assignments.

**4. CI/CD Pipeline**

- The CI/CD pipeline is the last remaining setup item before the team can start working on issues.
- Sam had not set it up yet. Itai clarified that in the context of this project it primarily means linting, similar to what was done in the course labs where branches had to pass checks before merging.
- Custom unit tests (e.g., expected value checks) can be added later once the codebase has something to test against.

**5. Prototypes Folder & Baseline Codebase**

- Sam organized all individual member prototypes into a prototypes folder in the repo. The prototype submitted to Ayla is also included there.
- Sam deliberately did not use any of the existing prototypes as the starting point for the main codebase. His reasoning: any prototype one-shotted with an LLM already implements too much, which would overlap with and muddy multiple Sprint 1 issues.
- Instead, he created a minimal skeleton: a basic index.html, style.css, and one JS file in the /src directory. It does not run as a game yet, which is intentional. The project is meant to grow cleanly through the issues.
- File structure clarified: all JS files live in /src, the majority of code goes there, while index.html and style.css sit at the root level.

**6. Nishant's Prototype Demo**

- Nishant pushed a new prototype to the repo just before the meeting — a playable proof of concept for his "build a website" idea, where the player types through HTML, then CSS, then JS in sequence.
- Sam ran it live. The game required typing HTML tags, enforced capitalization, and had a health system. Several bugs were found live: backspace killed the player, tab vs. space input handling was inconsistent.
- Despite the bugs, the group was enthusiastic about the concept. Sam noted it validated the post-MVP idea well.
- Itai proposed integrating it as the boss-level mechanic: each boss encounter could require the player to type a chunk of website code, building toward a rendered site.
- Open question raised: how short can the website snippets be while still producing something worth rendering? The group agreed the length is customizable and not a blocker.
- General consensus: the concept slots into the existing game structure cleanly and is worth pursuing once the core MVP sprints are complete.
